# AIエージェントのための設計原則

この文書は、AIエージェントがこのリポジトリで設計判断を行うときの基準を定める。
目的は、抽象的に「きれいな設計」を目指すことではない。
既存の小さな公開インターフェイス、薄い起動ファイル、責務が分かれた内部モジュールに沿って、必要十分な変更を選ぶことである。

設計判断では、まず現在の問題を解くために必要な情報、ルール、変更理由を確認する。
そのうえで、外に見せる約束と、内部で局所的に扱えばよい都合を分ける。
概念、型、関数、レイヤーを増やすのは、それによってコードの意味、制約、変更範囲、失敗時の扱いが明確になる場合だけにする。

## 1. 問題に必要なモデルだけを作る

良いモデルは、現在の問題を解くために必要な情報、ルール、判断だけで構成される。
自然に見える概念でも、今の変更に使わないなら最初は含めない。

良い判断:

- 既存の state、fs helper、render span で表現できるなら、それを使う。
- 同じ意味の値は同じ形で扱い、違う意味の値は混ぜない。
- 重要な制約やユーザー可視のルールには、名前や関数として明確な形を与える。

避ける判断:

- 将来使うかもしれないという理由だけで分類、型、中間データ構造を増やす。
- 1か所の処理を説明するためだけに、新しい概念を公開モデルへ持ち込む。
- 実装都合の名前を、問題領域の意味であるかのように扱う。

## 2. 公開インターフェイスは利用側の約束だけに絞る

公開インターフェイスは、モジュールを使う側が知る必要のある約束である。
何を受け取り、何を返し、何を変更し、どのような場合に失敗するのかを自然に理解できる形にする。

良い判断:

- 新しい公開関数、コマンド、`g:v9filer_*` オプションを追加する前に、既存の公開APIで表現できるか確認する。
- 利用側が判断するために必要な制約は隠さない。
- 内部の処理順序、辞書構造、描画手順、ファイルシステム実装の都合は不要に外へ出さない。

避ける判断:

- 内部実装を整えるためだけの helper や一時データを公開契約にする。
- 呼び出し順序、暗黙の状態、隠れた副作用に強く依存するAPIを作る。
- 既存の薄い entry point にロジックを増やす。

## 3. 安定した考え方を変わりやすい実装都合から守る

依存方向は、中心となるモデルやルールが、画面、バッファ、ファイルシステム、外部ツール、フレームワークの都合に振り回されないように決める。
変わりやすいものが、変わりにくいものに従う形にする。

良い判断:

- `plugin/` は起動と登録に留め、実際の処理は `autoload/` 側へ置く。
- 状態の読み書きは既存の state helper を通し、内部表現へ直接依存しない。
- 描画、ファイル操作、状態管理の責務を混ぜない。

避ける判断:

- UI上の都合を、ファイル操作や状態モデルの中心ルールとして扱う。
- ある実装の都合に合わせて、無関係なモジュールの責務を広げる。
- 小さな変更のために依存方向を逆流させる。

## 4. 内部実装は局所的で直接的に保つ

公開されるモデルや境界は慎重に設計する。
一方で、境界の内側では、局所的に読めて安全な直接実装を優先する。

良い判断:

- 小さな条件分岐、一時変数、短い helper で十分なら、そのままにする。
- 分割は、責任、変更理由、テスト単位を分ける必要があるときに行う。
- 変更に必要な情報が近い場所にあり、処理の流れを追いやすい形にする。

避ける判断:

- 関数やレイヤーを増やすこと自体を読みやすさと考える。
- 直接比較すれば読める条件を、意味のない helper 名で隠す。
- 外から見える振る舞いも重要なルールも変わらない抽象化を追加する。

## 設計前チェック

設計や実装に入る前に、次を確認する。

- 既存の関数、状態、helper、公開APIで表現できるか。
- 追加する概念や抽象化は、公開契約、重要なルール、変更影響の局所化、テスト容易性、失敗時の扱いやすさのどれを改善するか。
- ユーザーから観測できる挙動が変わる場合、`doc/v9filer.txt`、テスト観点、既存デフォルトへの影響を確認したか。
- その変更は、このリポジトリの既存の依存方向と責務分担に沿っているか。
